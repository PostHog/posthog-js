- Item: issue #3814 — User feedback support
- Conclusion: Valid React Native feature request; current RN surveys do not appear to implement feedback widget/user feedback support.
- Labels: enhancement, react-native, feature/mobile, feature/surveys
- URL: https://github.com/PostHog/posthog-js/issues/3814
- Relevant files: `packages/react-native/src/surveys/PostHogSurveyProvider.tsx`, `packages/react-native/src/surveys/components/Surveys.tsx`, `packages/react-native/src/surveys/getActiveMatchingSurveys.ts`, `packages/react-native/src/surveys/index.ts`, `packages/react-native/src/index.ts`, `packages/browser/src/extensions/surveys.tsx`, `packages/core/src/types.ts`, `packages/react-native/CHANGELOG.md`
- Findings: `packages/core/src/types.ts` defines `SurveyType.Widget`, so the shared survey model includes feedback/widget-style surveys.; `packages/browser/src/extensions/surveys.tsx` implements `FeedbackWidget` and handles `SurveyType.Widget` display logic for tab and selector widgets.; `packages/react-native/src/surveys/PostHogSurveyProvider.tsx` loads surveys and renders `SurveyModal`, but `shouldShowModal` currently only allows `SurveyType.Popover`.; `PostHogSurveyProvider.tsx` contains a commented-out `useFeedbackSurvey` hook that looks intended to find `SurveyType.Widget` surveys by `appearance.widgetSelector`, but it is not active or exported.; `packages/react-native/src/surveys/index.ts` exports `PostHogSurveyProvider`, `SurveyModal`, and `Questions`, but does not export a feedback-specific hook/component.; `packages/react-native/CHANGELOG.md` shows RN survey support exists, but I did not find evidence that feedback widget/user feedback support specifically is implemented.
- Fix assessment: A small code path is hinted at by the commented hook, but the issue does not define the intended React Native UX/API. Web widget semantics like tab/selector do not translate directly to native apps, so implementing without product direction risks adding the wrong public API.

## 2026-07-06T09:44:17.490Z
- Item: issue #3813 — Stop all integrations like session replay and autocapture on opt-out
- Conclusion: Valid React Native enhancement/privacy issue; opt-out currently persists consent and drops events, but does not stop integration work like session replay or autocapture hooks.
- Labels: enhancement, react-native, feature/mobile, feature/replay, feature/autocapture
- URL: https://github.com/PostHog/posthog-js/issues/3813
- Relevant files: `packages/react-native/src/posthog-rn.ts`, `packages/core/src/posthog-core-stateless.ts`, `packages/react-native/src/PostHogProvider.tsx`, `packages/react-native/src/autocapture.tsx`, `packages/react-native/src/hooks/useNavigationTracker.ts`, `packages/react-native/src/frameworks/wix-navigation.ts`, `packages/react-native/src/error-tracking/index.ts`, `packages/react-native/src/error-tracking/utils.ts`, `packages/react-native/src/optional/OptionalPlugin.ts`, `packages/react-native/test/posthog.spec.ts`, `packages/react-native/test/session-replay-rearm.spec.ts`
- Findings: `packages/react-native/src/posthog-rn.ts` overrides `optIn()` and `optOut()`, but both only call the core implementation and wait for storage persistence; there are no session replay/autocapture start or stop calls.; `packages/core/src/posthog-core-stateless.ts` stores `PostHogPersistedProperty.OptedOut` and drops queued/immediate events when `optedOut` is true, so integration-generated events are not sent while opted out.; RN session replay enablement uses `_isEnableSessionReplay()` as `!this.isDisabled && enableSessionReplay`; it does not check `optedOut`. Startup calls `startSessionReplay()` after initialization without an opt-out gate.; Native error tracking enablement similarly checks `isDisabled` but not `optedOut` in `_isAutocaptureNativeErrors()`.; Existing replay controls are available: `_stopSessionRecording()` calls `OptionalReactNativePlugin.stopRecording()`, linked-flag re-evaluation already pauses/resumes recording, and `startSessionRecording(false)` creates a fresh session by resetting the session id.; `PostHogProvider.tsx`, `autocapture.tsx`, and `useNavigationTracker.ts` gate touch/screen autocapture on provider options, not `posthog.optedOut`; capture calls are later dropped by core.; `captureAppLifecycleEvents()` registers an `AppState` listener that continues calling `capture()` for lifecycle events; it does not store an unsubscribe or check `optedOut` before calling capture.; `frameworks/wix-navigation.ts` registers a screen listener without retaining an unsubscribe and does not check `optedOut` before calling `posthog.screen()`.; Error-tracking JS wrappers in `error-tracking/utils.ts` do not return unsubscribe handles, and `OptionalPlugin.ts` exposes replay stop/start methods but not a general native plugin stop method.
- Fix assessment: A small replay-only change would be possible, but the issue asks for all integrations. RN currently has no shared integration lifecycle manager, several JS listeners are not removable, and native crash/error autocapture may need product/API decisions because the optional native plugin has no general stop method.

## 2026-07-06T09:45:52.794Z
- Item: issue #3811 — Rage touch detection
- Conclusion: Valid React Native feature request; RN touch autocapture exists but there is no rage-touch/rageclick detection today.
- Labels: enhancement, react-native, feature/mobile, feature/autocapture, feature/product-analytics
- URL: https://github.com/PostHog/posthog-js/issues/3811
- Relevant files: `packages/react-native/src/PostHogProvider.tsx`, `packages/react-native/src/autocapture.tsx`, `packages/react-native/src/types.ts`, `packages/react-native/test/autocapture.spec.ts`, `packages/browser/src/extensions/rageclick.ts`, `packages/browser/src/autocapture.ts`, `packages/core/src/posthog-core.ts`, `packages/types/src/posthog-config.ts`
- Findings: `packages/react-native/src/PostHogProvider.tsx` only wires touch autocapture through `onTouchEndCapture` when `captureTouches` is enabled, and contains a TODO noting it currently captures touch ends rather than confirmed presses.; `packages/react-native/src/autocapture.tsx` builds the RN element chain and sends `posthog.autocapture('touch', ...)` with `$touch_x` and `$touch_y`; it has no stateful repeated-touch/rage detection.; `packages/react-native/src/types.ts` defines `PostHogAutocaptureOptions` for touch capture, labels, no-capture props, screens, and navigation, but no rage-touch/rageclick option.; `packages/core/src/posthog-core.ts` shows `autocapture()` always enqueues event `$autocapture` with `$event_type`, so RN cannot emit `$rageclick` through that helper as-is.; `packages/browser/src/extensions/rageclick.ts` implements web rage click detection using default thresholds of 30px, 3 clicks, and 1000ms.; `packages/browser/src/autocapture.ts` uses the web `RageClick` detector to emit `$rageclick` during click autocapture, then continues with normal `$autocapture`.; `packages/react-native/test/autocapture.spec.ts` covers normal touch autocapture and no-capture behavior, but there are no tests for rage touch detection.
- Fix assessment: A small detector could be added, but this is a new mobile autocapture behavior and likely public API/event contract. The current RN code only captures touch-end events as `$autocapture`, while web emits a separate `$rageclick`; matching web semantics, thresholds, filtering, and gesture false-positive behavior needs product/API alignment first.
